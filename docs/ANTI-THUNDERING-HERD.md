# WebSocket 防雪崩重连 — 原理与实现说明

> 配套代码: `serialport-ws-client.js`

---

## 1. 什么是"雪崩式重连"

**场景**：服务端短暂宕机或重启，导致所有已连接的客户端同时断开。如果每个客户端立即重连或等固定间隔重连，大量请求会在同一瞬间涌入服务端，形成"惊群效应"（Thundering Herd），可能直接压垮刚恢复的服务。

```
时间轴:  ──────────────────────────────────────────────▶

服务端宕机 ──▶ 恢复上线

客户端A:  断线 ════════════════════════ 重连
客户端B:  断线 ════════════════════════ 重连       ← 同一时刻集中冲击！
客户端C:  断线 ════════════════════════ 重连
  ...
客户端N:  断线 ════════════════════════ 重连
```

---

## 2. 解决方案对比

| 方案 | 原理 | 防雪崩效果 | 恢复速度 |
|------|------|:---:|:---:|
| 固定延迟重连 | 断开后固定等 N 秒重试 | ❌ 差 — 仍可能同时冲击 | 快 |
| 防抖 (Debounce) | 连续触发只执行最后一次 | ❌ 差 — 可能永远不重连 | 不可控 |
| 节流 (Throttle) | 固定间隔最多执行一次 | ⚠️ 一般 — 时间窗口可能对齐 | 固定 |
| **指数退避 (Exponential Backoff)** | 每次失败等待时间翻倍 | ⚠️ 一般 — 不同客户端可能同步 | 慢 |
| **指数退避 + 随机抖动 (Exponential Backoff with Jitter)** ✅ | 翻倍 + 随机偏移 | ✅ 优秀 — 请求均匀打散 | 自适应 |

## 3. 核心算法：指数退避 + 随机抖动

### 3.1 计算公式

```
delay = min(base × 2^(attempt-1), maxDelay) × (1 + jitter × (random - 0.5))
```

| 参数 | 含义 | 推荐值 |
|------|------|--------|
| `base` | 基础延迟 | 1000ms |
| `attempt` | 当前重连次数（从 1 开始） | — |
| `maxDelay` | 延迟上限 | 30000ms |
| `jitter` | 抖动系数（0~1） | 0.3 |
| `random` | 随机数（0~1） | Math.random() |

### 3.2 重连时间线示例

以 `base=1000ms, maxDelay=30000ms, jitter=0.3` 为例：

```
第1次失败 → delay = min(1000 × 2⁰, 30000) × (1 + 0.3 × 随机) ≈ 850~1150ms
第2次失败 → delay = min(1000 × 2¹, 30000) × (1 + 0.3 × 随机) ≈ 1700~2300ms
第3次失败 → delay = min(1000 × 2², 30000) × (1 + 0.3 × 随机) ≈ 3400~4600ms
第4次失败 → delay = min(1000 × 2³, 30000) × (1 + 0.3 × 随机) ≈ 6800~9200ms
第5次失败 → delay = min(1000 × 2⁴, 30000) × (1 + 0.3 × 随机) ≈ 13600~18400ms
第6次失败 → delay = min(1000 × 2⁵, 30000) × (1 + 0.3 × 随机) ≈ 25500~30000ms
第7次失败 → delay = min(1000 × 2⁶, 30000) × (1 + 0.3 × 随机) ≈ 25500~30000ms
  ...
第N次   → delay = 30000ms (±15%)，达到上限封顶
```

## 4. 为什么节流/防抖不适合重连场景

### 4.1 防抖 (Debounce) 的问题

```javascript
// 防抖: 连续触发只执行最后一次
function debounce(fn, delay) {
  let timer;
  return () => {
    clearTimeout(timer);       // ← 每次断开都重置计时器
    timer = setTimeout(fn, delay);
  };
}
```

如果服务端**持续不可用**，每次心跳超时都会触发 `close` 事件 → 重置防抖计时器 → **永远不重连**。

### 4.2 节流 (Throttle) 的问题

```javascript
// 节流: 固定间隔最多执行一次
function throttle(fn, interval) {
  let lastTime = 0;
  return () => {
    const now = Date.now();
    if (now - lastTime >= interval) {
      fn();
      lastTime = now;
    }
  };
}
```

1000 个客户端同时断线，如果它们启动时间相近，"上次执行时间"相近，节流窗口可能完全对齐 → **依然集中冲击**。

## 5. 随机抖动的关键作用

### 无抖动 vs 有抖动对比

```
假设 1000 个客户端同时断线，第 3 次重连（理论延迟 4000ms）:

无抖动:
  ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓  ← 全部在 4000ms 同时重连
  ↑ 服务端瞬间承受 1000 次连接

有抖动 (jitter=0.3):
  ▓ ▓ ▓ ▓ ▓ ▓▓ ▓ ▓ ▓ ▓▓ ▓ ▓▓ ▓ ▓ ▓ ▓ ▓ ▓ ▓ ▓▓ ▓▓ ▓ ▓  ← 分散在 3400~4600ms
  ↑ 服务端平均每秒承受 ~800 次连接，峰值可控
```

### 1000 客户端重连分布模拟

```
连接数

  │                     ╱‾‾‾‾‾‾‾‾‾‾‾‾‾‾╲
  │                   ╱                  ╲
  │                 ╱                      ╲
  │               ╱                          ╲
```

抖动将集中的尖峰**摊平为均匀分布**，避免瞬时冲击。

## 6. 代码实现

```javascript
_scheduleReconnect() {
  // 检查是否超过最大重连次数
  if (this._maxReconnectAttempts > 0 && this._reconnectAttempt >= this._maxReconnectAttempts) {
    this._emit('reconnect_failed', { attempts: this._reconnectAttempt });
    return;
  }

  this._reconnectAttempt++;

  // 指数退避
  const base = this._reconnectBaseMs * Math.pow(2, this._reconnectAttempt - 1);

  // 封顶
  const capped = Math.min(base, this._reconnectMaxMs);

  // 随机抖动
  const jitter = capped * this._reconnectJitter * (Math.random() - 0.5);

  // 最终延迟
  const delay = Math.round(capped + jitter);

  this._emit('reconnecting', {
    attempt: this._reconnectAttempt,
    delay,
    maxAttempts: this._maxReconnectAttempts,
  });

  this._reconnectTimer = setTimeout(() => {
    this._reconnectTimer = null;
    this.connect().catch(() => { /* 失败会再次触发 _scheduleReconnect */ });
  }, delay);
}
```

## 7. 配置建议

| 场景 | base | maxDelay | jitter | maxAttempts |
|------|------|----------|--------|-------------|
| 局域网 / 同机部署 | 500ms | 10000ms | 0.2 | 0 (无限) |
| 广域网 / 生产环境 | 1000ms | 30000ms | 0.3 | 0 (无限) |
| 关键任务 / 快速恢复 | 200ms | 5000ms | 0.1 | 100 |
| 后台轮询 / 低优先级 | 5000ms | 120000ms | 0.5 | 0 (无限) |

- **base** 越大，初始等待越久，对服务端压力越小
- **maxDelay** 决定最长等待时间，避免重连间隔无限增长
- **jitter** 越大，分布越散，但个别客户端可能等更久
- **maxAttempts=0** 表示无限重连，适用于需要长期在线的场景

---

## 8. 总结

| 机制 | 解决什么问题 |
|------|------------|
| 指数退避 | 重连频率随失败次数递减，避免持续高频重试 |
| 随机抖动 | 多客户端重连时间均匀打散，避免同时冲击 |
| 封顶上限 | 保证重连间隔不会无限增长，确保最终恢复 |
| 主动断开标记 | 区分"用户主动断开"和"异常断开"，前者不触发重连 |

这套方案源自 Google、AWS 的云服务最佳实践，是处理分布式系统中重连/重试的标准方法。
